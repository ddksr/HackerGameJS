Guessing using personal information

If you have any information about the victim it can really help you guess the correct password. It is very common that people use dates (birthdays or anniversaries), their children or partners or pets names, favorite music artists etc. You can try victims date of birth, children names, wife or husband name, children names in combination, his first pet name. If a web pages enables hints it can be even easier to guess the hint and not the password.

Some hints

People often want to be tricky and add some simple numbers because they think that would help. So, if the targets wife is Sarah, the passwords could be: Sarrah, sarrah, sarrah123, sarrah1980 where 1980 is the year of Sarah's birth.

They also connect different words together. Imagine Simon has two kids, Philip and Britney. His password could be: philip, philipbritneybritneyphilip, pb, philipbritney123 ... Also, never forget about pets! If the target has two pets and he likes both equally, he will probably use both for his password. And add some numbers. It will often work.

People often use birthdays for their passwords. Lets say that Nick was born on 16th of September 1995. His password could be 09161995, 09/16/1995, 09/16, 1995, 1995-09-16, 1995-09-16, 1995-9-16 ...

Exploring database dumps

What is a database? A database is a piece of software that enables you to store any kind of data. Web pages use databases to store data about users, their actions etc.

Hackers can successfully steal some information from the databases. This is usually called a database dump. It can happen to anyone. If the web page stored passwords in the database, you can easily obtain it.

That is why it is never safe to store user's password. And almost nobody does it (except really stupid developers who should do something else). But how can a page remember a password and ask the user for the password without storing the password?

The solution: hashes. You change the password into something that cannot be reverted back. There are many hash functions available, some more safe and some less. One that is popular but unsafe is MD5. MD5 Example hashes: 

password -> 5f4dcc3b5aa765d61d8327deb882cf99
paSSword -> 7ec985d42fb6059a00418f1d5ed3f416
apple -> 1f3870be274f6c49b3e31a0c6728957f

But why is MD5 unsafe? There is no programmatic or mathematical way to change the hash back to the password. Try to google it: '5f4dcc3b5aa765d61d8327deb882cf99'. You will see that everyone knows this hash belongs to 'password'.

What does the web page do? The web page stores the hash of the password. On the 'Sign in' page, user types his or hers password, the page hashes the password and checks if this hash exists in its database. If it does, the user can log in, else the page knows that the password is incorrect. Look at the figures bellow to better understand what is going on.
